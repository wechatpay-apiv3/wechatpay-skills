# 业务逻辑完整性检查

> 本文档适用于品牌直连和服务商两种接入模式，包含两部分：
> 1. **必接接口清单** — 发券核销流程中必须接入的接口，缺少任何一个都可能导致流程不完整
> 2. **业务规则完整性检查** — 静态代码扫描时逐项检查的 7 项业务规则，每项必须独立判定（通过/未实现/不涉及）

---

## 第一部分：必接接口清单

以下接口必须接入，缺少任何一个都可能导致发券核销流程不完整。

### 设置商品券事件通知地址

接收回调的前提，必须先设置通知地址。

- 品牌：`POST /brand/marketing/product-coupon/notify-config`
- 服务商：`POST /v3/marketing/partner/product-coupon/notify-configs`

### 创建商品券

- 品牌：`POST /brand/marketing/product-coupon/product-coupons`
- 服务商：`POST /v3/marketing/partner/product-coupon/product-coupons`

### 商品券回调通知

接收发券/核销/退券事件，回调验签解密规范见 [📄 回调处理.md](./回调处理.md)

- 品牌直连 → [📄 品牌/签名验签/签名与验签规则.md](../../1-品牌/签名验签/签名与验签规则.md)
- 服务商 → [📄 服务商/签名验签/签名与验签规则.md](../../2-服务商/签名验签/签名与验签规则.md)

### 确认发放用户商品券

收到发券回调后必须调用，否则券不生效。

- 品牌：`POST /brand/marketing/product-coupon/product-coupons/{product_coupon_id}/users/{openid}/coupons/{code}/confirm`
- 服务商：`POST /v3/marketing/partner/product-coupon/users/{openid}/coupons/{coupon_code}/confirm`

### 核销用户商品券

- 品牌：`POST /brand/marketing/product-coupon/product-coupons/{product_coupon_id}/users/{openid}/coupons/{code}/use`
- 服务商：`POST /v3/marketing/partner/product-coupon/users/{openid}/coupons/{coupon_code}/use`

### 查询用户商品券详情

- 品牌：`GET /brand/marketing/product-coupon/product-coupons/{product_coupon_id}/users/{openid}/coupons/{code}`
- 服务商：`GET /v3/marketing/partner/product-coupon/users/{openid}/coupons/{coupon_code}`

### 选接接口

**向用户发放商品券（预发放）** — 仅在商户主动塞券场景下使用

- 品牌：`POST /brand/marketing/product-coupon/product-coupons/{product_coupon_id}/users/{openid}/pre-send-coupon`
- 服务商：`POST /v3/marketing/partner/product-coupon/users/{openid}/coupons`
- ⚠️ 摇优惠会场领券场景不需要调用此接口，用户在会场领券后只需调用**确认发放**接口即可

**券Code管理（预上传券Code）** — 仅在品牌方使用自定义券码时接入

- 品牌方希望使用自己的券码（如已有会员系统券码、线下印刷券码等）时需要接入，发券前先调用此接口上传自定义 Code，Code 不足会影响发券，需及时补充
- 如果没有自定义券码的需求，创建批次时选择微信支付自动生成券码即可，无需接入此接口

### 完整 API 路径

- 品牌 → [📄 品牌/接入指南/API接口清单.md](../../1-品牌/接入指南/API接口清单.md)
- 服务商 → [📄 服务商/接入指南/API接口清单.md](../../2-服务商/接入指南/API接口清单.md)

## 第二部分：业务规则完整性检查

> **静态代码扫描重点检查项**：扫描用户代码时，必须逐项检查以下 7 项是否实现。每一项必须独立执行搜索并独立给出判定（通过/未实现/不涉及），禁止将多项合并为一条结论——
> 1. 收到发券回调后是否调用了确认发放接口（第一项）
> 2. coupon_code_mode 为 UPLOAD 时是否调用了预上传券 Code 接口（第二项）
> 3. image_url 是否来自图片上传 API 而非外部链接（第三项）
> 4. store_scope 为 SPECIFIC 时是否调用了关联门店接口（第四项）
> 5. 核销券后是否调用了核销用户商品券接口通知微信（第五项）
> 6. 失效券后是否调用了失效用户商品券接口通知微信（第六项）
> 7. 退券后是否调用了退券接口通知微信（第七项）

### 一、发券确认

收到 `PRODUCT_COUPON.SEND` 回调后**必须调用确认发放接口**，否则券不生效。这是最常见的接入遗漏问题。

#### 确认发放接口路径

| 接入模式 | 接口路径 |
|---------|---------|
| 品牌直连 | `POST /brand/marketing/product-coupon/users/{openid}/coupons/{coupon_code}/confirm` |
| 服务商 | `POST /v3/marketing/partner/product-coupon/users/{openid}/coupons/{coupon_code}/confirm` |

#### 静态扫描检查方式

分两步搜索：
1. **搜索触发条件**：在整个项目中搜索 `PRODUCT_COUPON.SEND` 字符串，确认用户代码中是否存在发券回调分支
2. **搜索接口调用**：在整个项目中搜索确认发放接口路径关键字（品牌直连搜 `confirm`、服务商搜 `confirm`），确认是否存在对确认发放接口的调用

**判定规则**：如果步骤 1 找到了发券回调分支，但步骤 2 在整个项目中未找到确认发放接口的调用，则判定为接入遗漏。注意：不要只看回调分支内部的代码是否为空，而要搜索整个项目。

### 二、创建商品券批次 — 预上传券 Code

创券时 `stock.coupon_code_mode` 选择 `UPLOAD`（品牌方预上传 Code）时，品牌方需先调用**预上传券 Code 接口**将自定义 Code 上传到微信支付系统。

- 微信支付系统发券时从预存 Code 池中**随机选取**；当预存 Code 不足时**可能影响发券**，品牌应及时补充。
- 可通过应答中 `stock.coupon_code_count_info` 的 `available_count` 字段监控剩余可用 Code 数量。
- 此模式的商品批次**只可由品牌方自行发券**（即通过 API 发券），无法在"摇一摇有优惠"等微信支付渠道投放。

#### 静态扫描检查方式

如果用户代码中创建批次请求的 `coupon_code_mode` 值为 `UPLOAD`，检查是否存在对预上传券 Code 接口的调用。若缺失则提示：创券模式为 UPLOAD 时必须先调用预上传券 Code 接口，否则发券时无可用 Code。

### 三、商品券图片 — image_url 来源限制

`display_info.image_url`（商品券图片）**仅支持通过【图片上传 API】获取的图片 URL 地址**，不可使用外部链接或其他云存储地址。

#### 静态扫描检查方式

检查用户代码中设置 `image_url` 的值是否来自图片上传 API 的返回结果。若直接硬编码了非微信支付域名的图片地址，则提示：image_url 仅支持通过图片上传 API 获取的 URL。

### 四、可用门店范围 — store_scope 为 SPECIFIC 时关联门店

创建批次时 `stock.store_scope` 选择 `SPECIFIC`（特定门店可用）时，品牌需调用**关联门店接口**将门店与该批次关联，关联后该批次仅在关联的门店可用。

#### 静态扫描检查方式

如果用户代码中创建批次请求的 `store_scope` 值为 `SPECIFIC`，检查是否存在对关联门店接口（`associate-stores`）的调用。若缺失则提示：门店范围为 SPECIFIC 时必须调用关联门店接口，否则该批次无可用门店。

### 五、核销券后通知微信

品牌在自有系统完成券核销后，**必须调用核销用户商品券接口通知微信支付**，使微信侧券状态同步变更为「已核销」。如果不通知，微信侧券仍为「待使用」状态，用户在微信卡包中看到的券状态与实际不一致。

#### 核销用户商品券接口路径

| 接入模式 | 接口路径 |
|---------|---------|
| 品牌直连 | `POST /brand/marketing/product-coupon/product-coupons/{product_coupon_id}/users/{openid}/coupons/{code}/use` |
| 服务商 | `POST /v3/marketing/partner/product-coupon/users/{openid}/coupons/{coupon_code}/use` |

#### 静态扫描检查方式

分两步搜索：
1. **搜索触发条件**：在整个项目中搜索 `PRODUCT_COUPON.USE` 字符串，确认用户代码中是否存在核销回调分支
2. **搜索接口调用**：在整个项目中搜索核销接口路径关键字（品牌直连路径含 `coupons/` + `/use`，服务商路径含 `coupons/` + `/use`），确认是否存在对核销用户商品券接口的调用

**判定规则**：如果步骤 1 找到了核销回调分支，但步骤 2 在整个项目中未找到核销接口的调用，则判定为接入遗漏。注意：不要只看回调分支内部的代码是否为空，而要搜索整个项目。

### 六、失效券后通知微信

品牌将用户商品券失效后，**必须调用失效用户商品券接口通知微信支付**，使微信侧券状态同步变更为「已失效」。如果不通知，微信侧券仍保持原状态，用户可能误以为券仍可使用。

#### 失效用户商品券接口路径

| 接入模式 | 接口路径 |
|---------|---------|
| 品牌直连 | `POST /brand/marketing/product-coupon/users/{openid}/coupons/{coupon_code}/deactivate` |
| 服务商 | `POST /v3/marketing/partner/product-coupon/users/{openid}/coupons/{coupon_code}/deactivate` |

#### 静态扫描检查方式

分两步搜索：
1. **搜索触发条件**：在整个项目中搜索 `deactivate` 关键字，确认用户代码中是否存在失效相关的业务逻辑（如方法调用、接口定义或业务分支）
2. **搜索接口调用**：在整个项目中搜索失效接口路径关键字（路径含 `coupons/` + `/deactivate`），确认是否存在对失效用户商品券接口的调用

**判定规则**：如果步骤 1 找到了失效相关业务逻辑，但步骤 2 在整个项目中未找到失效接口的调用，则判定为接入遗漏。注意：不要只看某个方法体内部的代码是否为空，而要搜索整个项目。

### 七、退券后通知微信

品牌完成退券后，**必须调用退券接口通知微信支付**，使微信侧券状态同步更新。如果不通知，微信侧不知道券已退回，可能影响用户重新领取和批次库存回收。

#### 退券接口路径

| 接入模式 | 接口路径 |
|---------|---------|
| 品牌直连 | `POST /brand/marketing/product-coupon/users/{openid}/coupons/{coupon_code}/return` |
| 服务商 | `POST /v3/marketing/partner/product-coupon/users/{openid}/coupons/{coupon_code}/return` |

#### 静态扫描检查方式

分两步搜索：
1. **搜索触发条件**：在整个项目中搜索 `PRODUCT_COUPON.RETURN` 字符串，确认用户代码中是否存在退券回调分支
2. **搜索接口调用**：在整个项目中搜索退券接口路径关键字（品牌直连路径含 `coupons/` + `/return`，服务商路径含 `coupons/` + `/return`），确认是否存在对退券接口的调用

**判定规则**：如果步骤 1 找到了退券回调分支，但步骤 2 在整个项目中未找到退券接口的调用，则判定为接入遗漏。注意：不要只看回调分支内部的代码是否为空，而要搜索整个项目。
