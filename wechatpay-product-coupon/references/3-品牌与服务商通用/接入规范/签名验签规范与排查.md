# 公共 — 签名验签规范与排查

> 本文档适用于品牌直连和服务商两种接入模式。
> **注意：排查过程中无需提供私钥文件**，通过核对签名流程即可定位问题。
>
> **静态代码扫描重点检查项**：扫描用户代码时，必须逐项检查以下 4 项是否实现。每一项必须独立执行搜索并独立给出判定（通过/未实现/不涉及），禁止将多项合并为一条结论——
> 1. 是否对 HTTP 响应做了验签（第一章）
> 2. 是否对回调通知做了验签（第二章）
> 3. 是否实现了幂等去重和并发锁控制（第三章）
> 4. （可选）是否未对探测流量做特殊跳过处理（第四章）

## 一、签名方案与工具类差异

品牌直连和服务商使用不同的签名方案和工具类，**不能混用**：

| 项目 | 品牌直连 | 服务商 |
|------|----------|--------|
| 签名方案 | `WECHATPAY-BRAND-SHA256-RSA2048` | `WECHATPAY2-SHA256-RSA2048` |
| Authorization 身份标识 | `brand_id="xxx"` | `mchid="xxx"` |
| 签名用的私钥 | 品牌API证书私钥 | 商户API证书私钥 |
| 验签用的公钥 | 微信支付公钥 | 微信支付公钥（推荐）或平台证书 |
| API路径前缀 | `/brand/` | `/v3/` |
| Java 工具类 | `WXPayBrandUtility` | `WXPayUtility` |
| Go 工具类 | `wxpay_brand_utility` | `wxpay_utility` |
| 配置对象 | `BrandConfig`（传入 `brand_id`） | `MchConfig`（传入 `mchid`） |

**最常见的错误**：用了服务商的 `WXPayUtility` 去调品牌直连接口，或签名方案写错（漏了 `BRAND`），都会导致 401 SIGN_ERROR。

## 二、响应验签与回调验签

### 响应验签

接收微信支付 HTTP 响应时，商户**必须对响应进行签名验证**，确认响应确实来自微信支付，防止中间人篡改。

**验签步骤**：

1. 从响应 HTTP 头中获取 `Wechatpay-Timestamp`、`Wechatpay-Nonce`、`Wechatpay-Signature`、`Wechatpay-Serial`
2. 构造验签串（3 行，每行以 `\n` 结尾）：

```
应答时间戳\n
应答随机串\n
应答报文主体\n
```

3. 使用对应的公钥对验签串和签名值进行 SHA256 with RSA 验证
4. 验证 `Wechatpay-Serial` 与本地持有的公钥 ID / 证书序列号是否匹配

**静态扫描检查方式**：

分两步搜索：
1. **搜索 HTTP 客户端代码**：定位发送 HTTP 请求并处理响应的代码（如 OkHttp 的 `execute()`、HttpClient 的 `send()` 等）
2. **搜索验签调用**：在处理 2XX 成功响应的代码分支中，检查是否调用了响应验签方法（如 `validateResponse`、`verify` 等，或直接实现了上述验签步骤）

**判定规则**：如果 HTTP 客户端在收到 2XX 响应后直接解析返回数据，中间没有任何验签逻辑，则判定为未实现响应验签。

### 回调通知验签

商户系统对于回调通知的内容**必须做签名验证**，并校验通知的信息是否与商户侧的信息一致，防止数据泄露导致出现"假通知"，造成资金损失。

回调通知的验签方式与响应验签一致，从回调请求的 HTTP 头中获取签名信息，构造验签串并验证。

**静态扫描检查方式**：

分两步搜索：
1. **搜索回调处理代码**：定位处理微信支付回调通知的代码（如包含 `PRODUCT_COUPON.SEND`/`USE`/`RETURN` 等事件类型字符串的代码）
2. **搜索验签调用**：在回调处理流程中，检查在解密业务数据**之前**是否调用了通知验签方法（如 `validateNotification`、`verify` 等）

**判定规则**：如果回调处理代码在收到通知后直接解密或解析业务数据，中间没有任何验签逻辑，则判定为未实现回调验签。特别注意：如果使用了 `parseNotification` 等封装方法，需检查该方法内部是否包含验签步骤，而非只做了解密。

## 三、幂等与并发控制

同样的通知可能会多次发送给商户系统。商户系统**必须能够正确处理重复的通知**。

### 推荐做法

1. 收到通知后，先检查对应业务数据的状态，判断该通知是否已经处理
2. 如果未处理，则进行业务处理
3. 如果已处理，则直接返回成功结果

### 并发控制

在对业务数据进行状态检查和处理之前，**必须采用数据锁进行并发控制**，以避免函数重入造成的数据混乱。

建议使用分布式锁（如 Redis 锁）或数据库行锁，锁的粒度为 `coupon_code` + `event_type`。

**静态扫描检查方式**：

在回调处理代码中搜索以下特征：
1. **幂等去重**：是否有按 `coupon_code` + `event_type`（或等价的唯一键）查询已处理记录的逻辑，如数据库查询、缓存查询等
2. **并发锁控制**：是否有加锁逻辑，如 Redis 分布式锁（`setnx`/`RedisLock`）、数据库行锁（`SELECT ... FOR UPDATE`）、`synchronized`、`ReentrantLock` 等

**判定规则**：如果回调处理代码中既没有去重查询逻辑也没有加锁逻辑，则判定为未实现幂等去重和并发控制。

## 四、探测流量处理

微信支付会定期向商户回调地址发送探测流量，用于验证商户系统的连通性和签名验证逻辑。

### 处理要求

- 商户系统**不应对探测流量进行特殊处理**，而应将其视为正常的应答或通知回调，对其签名进行验证
- 探测请求的签名值以 `WECHATPAY/SIGNTEST/` 为前缀，验签时需正确处理该前缀
- 排查问题时，可以通过签名值中的 `WECHATPAY/SIGNTEST/` 前缀快速判断是否为探测流量

### 注意事项

- 所有用于探测目的的签名值都会包含 `WECHATPAY/SIGNTEST/` 前缀
- 如果商户系统对探测流量返回错误，微信支付可能会认为商户回调地址不可用

## 五、签名串格式

签名串固定 5 行，每行以 `\n` 结尾（包括最后一行）：

```
HTTP请求方法\n
URL\n
请求时间戳\n
请求随机串\n
请求报文主体\n
```

### 自查要点

1. **必须严格 5 行**，缺行或多行都会导致签名不一致
2. **每行末尾必须有 `\n`**，包括最后一行空行
3. URL 是**去掉域名的绝对路径**（不含 `https://api.mch.weixin.qq.com`）
4. 请求报文主体为空时（如 GET 请求），该行为空字符串，仍需附加 `\n`

## 六、四种参数类型的签名构造

### 6.1 Path 参数（路径参数）

**特征**：URL 中包含 `{参数名}`，如 `/brand/marketing/product-coupon/product-coupons/{product_coupon_id}`

**构造规则**：URL 中的 `{参数名}` 需替换为实际值。

示例 — 查询商品券（GET，无请求体）：

```
GET\n
/brand/marketing/product-coupon/product-coupons/200000001\n
1554208460\n
593BEC0C930BF1AFEB40B4A08C8FB242\n
\n
```

**常见错误**：
- URL 中保留了 `{product_coupon_id}` 占位符未替换
- URL 末尾多了或少了 `/`

### 6.2 Body 参数（包体参数）

**特征**：POST/PUT/PATCH 请求，请求体为 JSON

**构造规则**：签名串的第 5 行为完整的请求体 JSON 字符串。**签名时的 body 和实际发送的 body 必须完全一致**（包括字段顺序、空格、换行）。

示例 — 入会组件预授权（POST）：

```
POST\n
/brand/card-member/pre-auth-tokens\n
1554208460\n
593BEC0C930BF1AFEB40B4A08C8FB242\n
{"card_id":"pbLatjvWOibDc5-TBnbUk1pD12o0","openid":"obLatjnx9gnqzS4myYGmLZ7LgLBA"}\n
```

**常见错误**：
- 签名时 body 放一行，但实际请求发送时 body 做了格式化（多行/加空格），导致签名不匹配
- 签名时用了格式化的 JSON，但请求发送时压缩成一行
- body 中字段顺序与签名时不一致

### 6.3 Query 参数（查询参数）

**特征**：GET 请求带 `?key=value&key2=value2`

**构造规则**：URL 必须包含完整的查询字符串（`?` 和所有参数）。如果参数值包含特殊字符，需先 URL Encode。

示例 — 查询用户在品牌下所有会员卡（GET）：

```
GET\n
/brand/card-member/user-cards?openid=obLatjnx9gnqzS4myYGmLZ7LgLBA&user_card_state=EFFECTIVE&offset=0&limit=20\n
1554208460\n
593BEC0C930BF1AFEB40B4A08C8FB242\n
\n
```

**常见错误**：
- URL 中漏掉了 Query 参数部分（只签了路径，没签查询字符串）
- 含特殊字符的参数值未做 URL Encode（如 JSON 格式的参数值）
- 参数顺序与实际请求不一致

### 6.4 图片上传接口

**特征**：POST，Content-Type 为 `multipart/form-data`

**构造规则**：签名串的第 5 行为 `meta` 字段的 JSON（包含 `filename` 和 `sha256`），**不是整个 multipart body**。

示例 — 营销图片上传：

```
POST\n
/brand/card-member/media/image-upload\n
1554208460\n
593BEC0C930BF1AFEB40B4A08C8FB242\n
{"filename":"wechatpay_logo.png","sha256":"d2973a45b1d528c21ebb77792ef3fcea40fa9a4e04a17e35369102ba9c84c8b1"}\n
```

**常见错误**：
- 用整个 multipart body 参与签名，而非 meta 的 JSON
- sha256 值计算错误（应为文件内容的 SHA256 摘要）
- HTTP 头未设置 `Content-Type: multipart/form-data`

## 七、Authorization 头格式

```
Authorization: WECHATPAY-BRAND-SHA256-RSA2048 brand_id="100XX",nonce_str="随机串",signature="签名值",timestamp="时间戳",serial_no="证书序列号"
```

### 自查要点

| 检查项 | 正确做法 |
|--------|----------|
| 认证类型 | 品牌直连用 `WECHATPAY-BRAND-SHA256-RSA2048`，服务商用 `WECHATPAY2-SHA256-RSA2048` |
| 身份标识 | 品牌直连用 `brand_id`，服务商用 `mchid` |
| nonce_str | 必须与签名串中的随机串一致 |
| timestamp | 必须与签名串中的时间戳一致，且不能太旧 |
| serial_no | 必须是签名所用私钥对应的证书序列号 |
| Wechatpay-Serial | 请求头中需传微信支付公钥ID（用于验签） |
| 整行不换行 | Authorization 值必须在一行，不能换行 |

## 八、完整自查清单

遇到 401 SIGN_ERROR 时，无需提供私钥文件，按以下顺序逐项确认：

1. **签名方案是否正确**：品牌直连是否用了 `WECHATPAY-BRAND-SHA256-RSA2048`？服务商是否用了 `WECHATPAY2-SHA256-RSA2048`？
2. **Authorization 身份标识是否正确**：品牌直连是否用 `brand_id`？服务商是否用 `mchid`？
3. **证书序列号是否匹配**：`serial_no` 是否是当前签名私钥对应的证书序列号？
4. **签名串是否为 5 行**：HTTP方法、URL、时间戳、随机串、请求体，每行末尾 `\n`。可打印签名串（不含私钥内容）核对格式
5. **URL 是否正确**：
   - 去掉域名部分
   - Path 参数是否已替换为实际值
   - Query 参数是否包含在 URL 中
6. **请求体是否一致**：签名时的 body 和实际请求发送的 body 是否完全一致（含字段顺序、空格、换行）
7. **时间戳和随机串是否一致**：签名串中的值和 Authorization 头中的值是否相同
8. **时间戳是否过期**：系统时间是否准确，时间戳是否为当前时间
9. **图片上传**（如适用）：签名串的 body 部分是否只用了 meta 的 JSON？
10. **以上都正确时**：参照上方各参数类型的签名构造示例，用测试公私钥验证签名计算逻辑
