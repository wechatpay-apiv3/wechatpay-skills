## 一、回调处理

### 前提条件

1. **必须设置 APIv3 密钥**（32字节），未设置不会收到任何回调
2. **必须调用「设置商品券事件通知地址」API** 配置 `notify_url`
   - 品牌直连：`POST /brand/marketing/product-coupon/notify-config`
   - 服务商：`POST /v3/marketing/product-coupon/notify-config`（服务商维度，所有品牌共用）
3. 回调地址要求：HTTPS + 域名已 ICP 备案 + 公网可访问
4. 不能使用内网地址（127.0.0.1 / 192.168.x.x / localhost）

### 回调 IP 白名单

| 出口 | 网段/IP |
|------|---------|
| 上海电信出口网段 | 101.226.103.0/25 |
| 上海联通出口网段 | 140.207.54.0/25 |
| 上海CAP出口网段 | 121.51.58.128/25 |
| 深圳电信出口网段 | 183.3.234.0/25 |
| 深圳联通出口网段 | 58.251.80.0/25 |
| 深圳CAP出口网段 | 121.51.30.128/25 |
| 香港出口网段 | 203.205.219.128/25 |
| 广州腾讯云出口IP | 81.71.199.64, 81.71.198.25, 81.71.199.59 |
| 退款结果通知、分账动账通知IP | 175.24.214.208, 175.24.211.24, 175.24.213.135, 109.244.180.23, 114.132.203.119, 43.139.43.69 |

同时关闭 WAF/CC 防护对回调 URL 的拦截，避免误将微信支付回调请求判定为恶意请求。

### 回调事件类型

| event_type | 说明 | 处理动作 |
|-----------|------|---------|
| PRODUCT_COUPON.SEND | 发券通知 | 记录 + **调用确认发放API** |
| PRODUCT_COUPON.USE | 核销通知 | 更新核销状态 |
| PRODUCT_COUPON.RETURN | 退券通知 | 更新退还状态 |

> ‼️ **收到 PRODUCT_COUPON.SEND 回调后必须调用确认发放接口，否则券不生效。** 这是最常见的接入遗漏问题。静态代码扫描时需检查用户代码中是否包含以下接口路径：
> - 品牌：`/brand/marketing/product-coupon/users/{openid}/coupons/{coupon_code}/confirm`
> - 服务商：`/v3/marketing/partner/product-coupon/users/{openid}/coupons/{coupon_code}/confirm`

### 回调报文与解密

```json
{
  "id": "通知唯一ID",
  "create_time": "2025-08-02T00:00:00+08:00",
  "event_type": "PRODUCT_COUPON.SEND",
  "resource_type": "encrypt-resource",
  "resource": {
    "algorithm": "AEAD_AES_256_GCM",
    "ciphertext": "Base64编码的密文",
    "nonce": "随机串",
    "associated_data": "product_coupon"
  }
}
```

- 算法: `AEAD_AES_256_GCM`，密钥: APIv3 密钥（32字节）
- 从 `resource` 取 `nonce`、`ciphertext`、`associated_data` 进行解密

### 解密后关键字段

`coupon_code`、`product_coupon_id`、`stock_id`、`openid`、`unionid`、`send_channel`、`send_request_no`、`receive_time`、`valid_begin_time`/`valid_end_time`

### 回调处理要求

1. **必须返回 HTTP 2XX**（200 或 204），否则微信支付会重试
2. **必须在 5 秒内应答**
3. **必须做幂等处理**（按 coupon_code + event_type 去重）
4. **必须验签**，防止伪造通知（品牌直连用微信支付公钥，服务商用微信支付平台证书）
5. 签名探测流量以 `WECHATPAY/SIGNTEST/` 开头，需正确处理
6. 即使业务处理异常，也建议返回 200，通过告警系统人工介入

### 回调收不到的常见排查场景

#### 一、前置配置缺失

1. **未设置 APIv3 密钥** — 微信支付不会发送回调通知
2. **未配置 `notify_url`** — 接口或商户平台未配置回调通知接收地址，微信支付不会发送回调通知

#### 二、回调地址配置类问题

1. **地址格式错误**
   - `notify_url` 未以 `https://` 或 `http://` 开头
   - URL 中只有域名，缺少具体路径（如 `http://www.weixin.qq.com`）
   - URL 携带了参数
   - 使用了内网地址（`127.0.0.1`、`192.168.x.x`、`localhost`）
2. **域名未备案或解析异常**
   - 域名未完成工信部 ICP 备案（国内服务器必须）
   - DNS 解析失效（解析记录过期、未配置正确的 A/AAAA 记录）

#### 三、网络与服务器连通性问题

1. **防火墙/安全组拦截** — 未对上方「回调 IP 白名单」中的 IP 段开通入站规则
2. **WAF/CC 防护误拦** — 安全策略将微信支付回调请求误判为恶意请求
3. **网络链路故障** — 丢包或延迟过高（超过 3 秒）导致请求超时
4. **CDN/反向代理配置异常** — Nginx、Cloudflare 等未将回调请求正确转发至后端服务

#### 四、回调处理逻辑问题

1. **登录态校验** — `notify_url` 的代码逻辑不能做登录态校验
2. **未在 5 秒内应答** — 微信支付会认为通知失败并重复发送
3. **未做幂等** — 同一通知可能多次发送，必须按 `coupon_code` + `event_type` 去重

### 服务商回调特殊说明

- 回调地址是**服务商维度**配置的，所有品牌共用同一个回调地址
- 通过回调内容中的 `brand_id` 区分不同品牌

## 二、错误处理策略

| 错误类型 | 处理策略 |
|---------|---------|
| 500 SYSTEM_ERROR | 使用相同 out_request_no 重试（指数退避） |
| 400 参数错误 | 修正参数后重试 |
| 401 签名错误 | 检查证书和签名方案是否与接入模式匹配 |
| 回调超时 | 返回200，异步补偿处理 |
| 解密失败 | 检查 APIv3 密钥是否正确 |

## 三、幂等设计

- 所有写操作使用唯一的 `out_request_no` / `send_request_no`
- 相同请求号重复请求不会创建重复资源
- 建议格式：`{业务前缀}_{日期}_{序号}`，如 `SEND_20250801_000001`

## 四、请求域名

- 主域名: `https://api.mch.weixin.qq.com`
- 备域名: `https://api2.mch.weixin.qq.com`
