# 品牌直连排障手册

> **错误码提取**：Request-ID 的格式为 `{请求标识}-{错误码}`，**截取最后一个 `-` 后面的数字即为错误码**。例如 `08E68E97CC0610DF0618AC86BCA80120082880CF05-268578704` → 错误码 `268578704`。
>
> **使用方式**：提取到错误码后，先在下方速查表中匹配错误码，命中后查找对应的详细排查章节。未命中则按「常见错误场景」兜底处理。
>
> ‼️ **示例代码推荐输出规则**：各错误码的「示例代码推荐」列出了 Java 和 Go 两种语言的示例，向用户输出时需先确认开发语言，**只输出匹配的示例**。**不要在排障流程一开始就问开发语言**，仅在推荐示例代码时确认即可。
>
> ‼️ **语气要求**：排障回复应亲和、专业，像一个有经验的技术支持在帮忙。避免冷冰冰地罗列信息或堆砌文档目录。用自然的对话语气解释问题原因和修复方案。

## 常见错误场景（兜底）

> 当错误码不在下方速查表中时，按以下场景分类排查。

**参数格式问题**：时间格式不是 RFC3339（正确：`2025-08-01T00:00:00+08:00`）、金额单位错误（应为分）、`out_request_no` 长度不在 6-40 字符、必填字段缺失 → 对照接口文档逐一校验。

**业务配置问题**："AppID非法" → 未通过交易连接名片关联品牌；"OpenID不合法" → OpenID 与 AppID 不匹配；"全场券/单品券配置错误" → 全场券优惠在 `single_usage_info`，单品券在 `stock.single_usage_rule`。

**签名验签问题**：品牌直连用 `WECHATPAY-BRAND-SHA256-RSA2048` + `brand_id="xxx"`。详细自查清单见 [📄 签名验签规范与排查.md](../../3-品牌与服务商通用/接入规范/签名验签规范与排查.md)。

**权限问题**：检查品牌经营平台权限，确认 brand_id 与当前身份匹配。

**频率限制**：降低请求频率，使用指数退避策略重试（1s → 2s → 4s）。

**服务端异常**：使用相同的 `out_request_no` / `send_request_no` 重试（幂等），指数退避，最多重试 3 次。

## 一、券状态与生命周期类

### 268578776 — 用户券已过期

**场景**：对已过期的券执行核销、退券等操作时触发。

**常见真实原因**：

1. **有效期起算时间不一致**：微信侧以**回调通知时间**作为券的生效起算时间，但商家侧以发券接口返回成功的时间作为生效时间。两边起算时间有差异时，商家侧认为券还在有效期内，但微信侧已判定过期。
   - 正确做法：**记录收到回调通知的时间**，确认发放时以回调时间为准计算有效期。
2. **确认发放延迟导致有效期缩短**：用户在微信侧摇到券后，商家因业务流程（如要求用户先注册会员）延迟调用确认发放接口。延迟期间有效期已在消耗，等确认发放时实际可用时间已所剩无几。
   - 典型场景：用户摇到券 → 商家要求注册会员 → 用户犹豫未注册 → 几天后注册并确认发放 → 券有效期已过大半或已过期。
3. **确认发放后商家侧单方面修改有效期**：确认发放时已将有效期同步给微信（如 3 天），之后商家侧在自己系统里改为 5 天，但微信侧仍按原 3 天计算，导致商家认为券有效但微信侧已过期。
   - 关键点：**确认发放后券的有效期无法修改**，微信侧没有修改券有效期的接口。

**排查步骤**：

> 📋 **排查前需要你提供以下信息**：
>
> **身份配置参数：**
> - **品牌 ID**（brand_id）
> - **API 证书序列号**（serial_no）
> - **微信支付公钥 ID**（wechat_pay_public_key_id）
> - **签名值（Base64）**+**时间戳**+**随机串**
>
> **业务参数：**
> - **券码**（coupon_code）—— 出问题的那张券
> - **用户 OpenID**（openid）
> - **商品券 ID**（product_coupon_id）
> - **批次 ID**（stock_id）
> - **AppID**（appid）

1. 收齐上述参数后，执行排障脚本 [`查询用户商品券详情_品牌.py`](../../scripts/查询用户商品券详情_品牌.py)（脚本支持本地签名和预签名两种模式，会自动处理签名、发送请求并结构化输出关键字段和诊断提示）。
2. 根据脚本输出的「券关键信息」和「自动诊断」确认过期原因：重点关注 `券状态`、`有效期结束` 时间、`领取时间` 与 `确认发放时间` 的间隔
3. 对比输出的时间信息与商家侧记录的有效期是否一致
4. 检查商家侧是否在确认发放后单方面修改了有效期配置

> 💡 **示例代码推荐**：排查此问题需要查询券状态，**必须主动向用户推荐以下示例代码**：
> - 查询用户商品券详情：
>   - Java 示例：`1-品牌/代码示例/Java/5-商品券查询/GetUserProductCoupon.java`
>   - Go 示例：`1-品牌/代码示例/Go/5-商品券查询/GetUserProductCoupon.go`

**修复建议**：

1. **当前这张过期券**：已是终态，无法再执行核销、退券等任何操作。业务侧应将该券状态同步为「已过期」，并向用户说明券已失效。
2. **预防后续券再出现意外过期**：
   - **有效期起算时间必须以回调通知时间为准**，在收到回调时记录时间戳，不要以发券接口返回时间作为起算点
   - **收到发券回调后尽快调用确认发放接口**，减少延迟导致的有效期损耗
   - **确认发放后不要在商家侧单方面修改有效期**，微信侧没有修改券有效期的接口，如需调整须创建新批次

### 268578367 — 用户券状态非法

**场景**：对券执行的操作与当前券状态不匹配。

**常见真实原因**：

1. **没调确认发放就直接核销**：收到发券回调后跳过了确认发放，直接调核销接口。此时券状态还是 `CONFIRMING`，核销要求 `EFFECTIVE`，状态不匹配报错。
2. **券已失效后还来调核销**：券已被失效（`DEACTIVATED`），商家侧没同步状态，还对该券发起核销。

**排查步骤**：
1. 调用「**查询用户商品券详情**」接口查看当前券状态。向用户收集排查所需的参数，然后执行排障脚本 [`查询用户商品券详情_品牌.py`](../../scripts/查询用户商品券详情_品牌.py)。
2. 根据脚本输出的 `券状态` 判断：如果是 `CONFIRMING`，说明缺少确认发放步骤
3. 如果是 `DEACTIVATED`，说明券已失效，商家侧需同步状态

> 💡 **示例代码推荐**：根据排查结果，用户可能需要接入以下接口，**必须主动向用户推荐以下示例代码**：
> - 查询用户商品券详情：
>   - Java 示例：`1-品牌/代码示例/Java/5-商品券查询/GetUserProductCoupon.java`
>   - Go 示例：`1-品牌/代码示例/Go/5-商品券查询/GetUserProductCoupon.go`
> - 确认发放用户商品券：
>   - Java 示例：`1-品牌/代码示例/Java/3-商品券发放/ConfirmSendUserProductCoupon.java`
>   - Go 示例：`1-品牌/代码示例/Go/3-商品券发放/ConfirmSendUserProductCoupon.go`
> - 核销用户商品券：
>   - Java 示例：`1-品牌/代码示例/Java/4-商品券核销/UseUserProductCoupon.java`
>   - Go 示例：`1-品牌/代码示例/Go/4-商品券核销/UseUserProductCoupon.go`

**修复建议**：
- 必须按 发券回调 → **确认发放** → 核销 的流程走，不能跳步
- 对已失效的券不再发起操作

### 272756745 — 券已被其他订单核销

**场景**：尝试核销一张已经被核销过的券。

**常见真实原因**：同一张券 code 多次调用核销接口，但每次请求使用了**不同的核销请求单号**（`out_request_no`）。微信侧判定为不同的核销请求，第一次成功后，后续请求因券已被核销而报错。

**排查步骤**：
1. 调用「**查询用户商品券详情**」接口确认券状态和核销信息。向用户收集排查所需的参数，然后执行排障脚本 [`查询用户商品券详情_品牌.py`](../../scripts/查询用户商品券详情_品牌.py)。
2. 确认脚本输出的券状态为 `USED`，查看核销信息中的 `transaction_id`、`out_trade_no`
3. 比对多次核销请求的 `out_request_no` 是否一致——**不一致是最常见的原因**

> 💡 **示例代码推荐**：排查和修复此问题需要查询券状态及正确核销，**必须主动向用户推荐以下示例代码**：
> - 核销用户商品券：
>   - Java 示例：`1-品牌/代码示例/Java/4-商品券核销/UseUserProductCoupon.java`
>   - Go 示例：`1-品牌/代码示例/Go/4-商品券核销/UseUserProductCoupon.go`
> - 查询用户商品券详情：
>   - Java 示例：`1-品牌/代码示例/Java/5-商品券查询/GetUserProductCoupon.java`
>   - Go 示例：`1-品牌/代码示例/Go/5-商品券查询/GetUserProductCoupon.go`

**修复建议**：
- **确保同一张券的核销请求单号（`out_request_no`）始终一致**，这样重复调用时微信会幂等处理，直接返回成功
- 建议 `out_request_no` 的生成规则包含券码，如 `USE_{coupon_code}_{order_id}`，天然保证同一张券的单号一致
- 如果确实是不同订单核销同一张券，说明券已被使用，业务侧需提示用户

### 272756746 — 券已被其他订单核销锁定

**场景**：券正在被另一个核销请求处理中（并发核销场景）。

**排查步骤**：
1. 确认是否存在并发核销请求（多个系统/线程同时对同一张券发起核销）
2. 检查核销请求的 `out_request_no` 是否不同

**修复建议**：
- 等待前一个核销请求完成后重试（建议间隔 1-2 秒）
- 业务侧应对同一张券的核销请求做串行化处理，避免并发
- 使用相同的 `out_request_no` 重试可保证幂等

### 272756769 — 券已被删除

**场景**：操作的券已被删除。

**常见真实原因**：

1. **用户在微信卡包中手动删除了券**：用户可以在微信卡包中主动删除券，目前**仅卡包路径可以触发删除**，且**没有回调通知机制**——微信不会向商户推送"券被用户删除"的通知。商户侧完全不知道券已被删除，再对该券发起操作时就会报错。

**排查步骤**：
1. 调用「**查询用户商品券详情**」接口确认券状态。向用户收集排查所需的参数，然后执行排障脚本 [`查询用户商品券详情_品牌.py`](../../scripts/查询用户商品券详情_品牌.py)。
2. 确认脚本输出的券状态为 `DELETED` 或 `DEACTIVATED`
3. 商户侧日志中通常找不到任何失效操作记录——因为删除动作发生在用户微信卡包侧

> 💡 **示例代码推荐**：排查此问题需要查询券状态，**必须主动向用户推荐以下示例代码**：
> - 查询用户商品券详情：
>   - Java 示例：`1-品牌/代码示例/Java/5-商品券查询/GetUserProductCoupon.java`
>   - Go 示例：`1-品牌/代码示例/Go/5-商品券查询/GetUserProductCoupon.go`
> - 失效用户商品券：
>   - Java 示例：`1-品牌/代码示例/Java/6-商品券失效与退券/DeactivateUserProductCoupon.java`
>   - Go 示例：`1-品牌/代码示例/Go/6-商品券失效与退券/DeactivateUserProductCoupon.go`

**修复建议**：
- 过期券已是终态，无需再调用失效接口，业务侧应在操作前判断券状态
- **在调用失效接口前，先检查券状态**，对已过期的券跳过操作即可

### 268578851 — 用户券不存在

**场景**：查询或操作的券码（`coupon_code`）在系统中不存在。

**常见真实原因**：

1. **确认发券时 `coupon_code` 传值不对**：没有取发券接口返回的 `coupon_code`，传了商家侧自己的券码或拼接错误的值。

**排查步骤**：
1. 对比确认发放请求中的 `coupon_code` 与发券接口返回的 `coupon_code`，确认是否一致
2. 注意大小写、前后空格等细节

**修复建议**：发券接口和确认发券接口的 `coupon_code` 必须一致，直接从发券返回值或回调通知中取原值透传。

### 272755717 — 券不在使用有效期（规则周期）

### 272755715 — 券不在使用有效期（绝对批次有效期）

3. 如果配置了 `wait_days_after_receive`（领取后等待N天生效），确认是否还在等待期内
4. 确认服务器时间是否准确（时区为 UTC+8）

> 💡 **示例代码推荐**：排查此问题需要查询券有效期信息，**必须主动向用户推荐以下示例代码**：

## 三、发券限制类

### 272756740 — 用户领取券张数已达上限

**场景**：用户领取同一批次的券数量已达到 `max_count_per_user` 的限制。最常见于私域塞券场景，创券限领 N 张但发券侧没做计数校验导致超发。

**排查步骤**：
1. 查批次配置的 `max_count_per_user`（必填，最多 100 次）
2. 查该用户在此批次下已有多少张券（含未确认发放的也计入名额）

>   - Java 示例：`1-品牌/代码示例/Java/5-商品券查询/ListUserProductCoupons.java`
>   - Go 示例：`1-品牌/代码示例/Go/5-商品券查询/ListUserProductCoupons.go`
> - 查询商品券批次详情：

>   - Go 示例：`1-品牌/代码示例/Go/2-商品券批次管理/GetStock.go`

**修复建议**：发券前先查用户已领张数，达上限不再发券。需要再发的话增大 `max_count_per_user` 或创新批次。

## 四、参数校验类

### 268578532 — 解析 OpenID 错误

**场景**：请求中的 `openid` 参数格式不合法或与 `appid` 不匹配。

**排查步骤**：
1. 确认 `openid` 格式是否正确（以 `o` 开头的 28 位字符串）
2. 确认 `openid` 是否对应请求中的 `appid`（OpenID 与 AppID 是一一对应的）
3. 确认 `appid` 是否已通过「交易连接名片」关联品牌
4. 检查是否混用了不同 AppID 下的 OpenID

**修复建议**：确保 `openid` 和 `appid` 来自同一个微信应用，且该 AppID 已关联品牌。

---

### 268435463 — 参数过短

**场景**：请求参数值长度低于接口要求的最小长度。

**排查步骤**：
1. 检查请求体中各字段值的长度
2. 常见过短字段：`out_request_no`（最少 6 位）、`name`（商品券名称不能为空）
3. 对照接口文档的字段长度要求逐一校验

**修复建议**：补全参数值至最小长度要求。

---

### 268435462 — 参数过长

**场景**：请求参数值长度超过接口要求的最大长度。

**排查步骤**：
1. 检查请求体中各字段值的长度
2. 常见过长字段：`out_request_no`（最长 40 位）、`name`（商品券名称有长度限制）、`usage_description`（使用说明有长度限制）
3. 对照接口文档的字段长度要求逐一校验

**修复建议**：截断或精简参数值至最大长度要求内。

---

### 268435461 — 参数非法

**场景**：请求参数值格式或内容不合法。

**排查步骤**：
1. 检查时间格式是否为 RFC3339（正确示例：`2025-08-01T00:00:00+08:00`）
2. 检查金额字段单位是否为**分**（非元）
3. 检查枚举值是否拼写正确（如 `ALL`/`SINGLE`、`NORMAL`/`DISCOUNT`/`EXCHANGE`）
4. 检查 URL 格式字段是否以 `https://` 开头
5. 检查是否有不可见字符（如全角空格、BOM 头等）

**修复建议**：对照接口文档逐一校验参数格式，特别注意时间格式和金额单位。

---

### 268435464 — 参数超出取值范围

**场景**：请求参数值超出了允许的数值范围。

**排查步骤**：
1. 检查金额类字段是否在合理范围内（如 `discount_amount` 不能大于 `threshold`）
2. 检查数量类字段（如 `max_count`、`max_count_per_user`）是否超出上限
3. 检查折扣类字段（如 `percent_off`）取值范围（通常 1-99）
4. 检查多次优惠的 `count` 是否在 3-15 范围内

**修复建议**：调整参数值至允许的范围内，对照接口文档确认各字段的取值约束。

## 五、业务配置类

### 268578704 — 传入的 AppID 非法

**场景**：请求中的 `appid` 未与品牌建立关联。

**排查步骤**：
1. 确认 `appid` 是否正确（注意大小写）
2. 确认该 AppID 是否已通过「交易连接名片」关联品牌：
   - 品牌直连：在品牌经营平台 UI 操作关联
3. 确认 AppID 类型是否支持（公众号 AppID / 小程序 AppID）

**修复建议**：先在品牌经营平台完成 AppID 与品牌的关联，再重试接口调用。参考 [📄 MAB关系说明文档.md](../../3-品牌与服务商通用/接入规范/MAB关系说明文档.md)。

### 268578608 — 当前访问对象不存在，请确认 ID 是否合法、属于当前品牌且由当前服务商创建

**场景**：请求中的资源 ID（如 `product_coupon_id`、`stock_id`）不存在或不属于当前品牌。

**排查步骤**：
1. 确认 `product_coupon_id` / `stock_id` 是否拼写正确
2. 确认该资源是否属于当前 `brand_id` 创建
3. 确认该资源是否已被失效或删除

**修复建议**：核实资源 ID 的来源，使用创建接口返回的 ID，避免手动拼接或硬编码。

### 268578404 — 缺少必要的商品券图片

**场景**：创建商品券时未设置必填的 `display_info.image_url` 字段。

**排查步骤**：
1. 检查创建商品券请求体中 `display_info.image_url` 是否已设置
2. 确认图片 URL 是否来自「图片上传 API」（仅支持通过上传接口获取的 URL）
3. 检查图片格式和大小是否符合要求（JPG/PNG，不超过 2M）

**修复建议**：先调用图片上传 API 获取图片 URL，再设置到 `display_info.image_url` 字段。不支持外部链接。

## 六、频率与系统类

### 270924289 — 频率过快

**场景**：接口调用频率超过限制。

**排查步骤**：
1. 检查短时间内是否有大量并发请求
2. 确认是否有重试逻辑导致请求放大（如无退避的循环重试）

**修复建议**：
- 降低请求频率，使用**指数退避策略**重试（第1次等1秒、第2次等2秒、第3次等4秒）
- 对批量操作做限流控制
- 使用相同的 `out_request_no` 重试可保证幂等，不会产生重复操作

## 七、创券配置类

### 268586067 — 非递减优惠每一轮的优惠力度应大于等于前一轮

**场景**：创建多次优惠券（`usage_mode=PROGRESSIVE_BUNDLE`）时，各轮优惠力度没有满足越来越便宜的要求。

**排查步骤**：检查各轮配置，每一轮对用户的优惠力度必须 ≥ 前一轮（即越来越便宜）。

**修复建议**：调整各轮优惠力度，确保后续轮次越来越优惠。
