## 一、接入流程

### 接入全流程（8步）

| 步骤 | 操作 | 操作方式 | 说明 |
|------|------|---------|------|
| 1. 品牌入驻 | API注册品牌，提交品牌资质审核 | **API** | 服务商代品牌方完成入驻 |
| 2. 创品牌门店 | API创建门店，等待平台审核 | **API**（可选，如有线下门店） | 有线下门店场景时需创建 |
| 3. 创交易连接名片 | API创建交易连接名片，关联支付场景 | **API** | 关联AppID的关键步骤，**不关联则后续创券报错** |
| 4. 创券 | API创建商品券，配置券面和规则 | **API** | 请求体中**必须传 `brand_id`**；配置券的scope/type/usage_mode等核心属性 |
| 5. 创批次/批次组 | API添加批次或批次组，设置发放数量和使用规则 | **API** | 设置库存、有效期、发放规则等 |
| 6. 创投放计划 | API创建投放计划，提交审核后生效 | **API** | 需人工审核，约2个工作日 |
| 7. 发券 | 服务商主动发券或用户主动领取，调用确认接口后券生效 | **API** | 预发放→token→领券→回调→确认发放 |
| 8. 核销 | 扫码核销或用户主动出示，验证后完成券核销 | **API** | 必须关联订单，券状态需为EFFECTIVE |

> **服务商优势**：全部 8 步都有 API，可实现全链路自动化对接。品牌直连的步骤1（入驻）、步骤3（交易连接名片）、步骤6（创活动/投放计划）没有对应API。

### 参数获取与签名配置（服务商平台）

服务商模式下，参数在**服务商平台**（pay.weixin.qq.com）获取，brand_id 由品牌方提供：

| 参数 | 获取路径 | 注意事项 |
|------|---------|---------|
| mchid（服务商商户号） | 服务商平台 → 账户中心 → 个人信息 | 所有接口必传，服务商唯一标识 |
| brand_id | 品牌方提供，或合作伙伴平台 → 品牌经营中心 → 合作品牌管理 | 服务商模式**必须在请求体传 `brand_id`** |
| 品牌授权 | 品牌方在品牌经营平台授予服务商「商品券」产品功能权限 | **未授权则调用 API 报 `NO_AUTH`** |
| 签名方案 | — | `WECHATPAY2-SHA256-RSA2048`，Authorization 头用 `mchid="xxx"` |
| 商户API证书序列号 | 服务商平台 → API安全 → 管理证书，或 `openssl x509 -in cert.pem -noout -serial` | 需与签名使用的商户API证书对应 |
| 商户API证书私钥文件 | 服务商平台 → API安全 → 申请证书 → 下载 `apiclient_key.pem` | **只能下载一次，丢失需重新申请；勿上传至 GitHub 等公开场所** |
| APIv3密钥 | 服务商平台 → API安全 → 设置密钥 | 32字节，**设置后无法查看，未设置则收不到回调** |
| 微信支付公钥ID | 服务商平台 → API安全 → 管理公钥 | — |
| 微信支付公钥文件 | 服务商平台 → API安全 → 微信支付公钥 → 申请后下载 | 用于**验签**，长期有效，无需定期更换 |
| appid | 服务商平台 → 产品中心 → APPID账号管理 → 我关联的APPID账号 | 未绑定需由超级管理员在 APPID账号管理中关联 |
| AppID关联品牌 | API `POST /v3/brand/partner/cardtemplate/traderelation` 或品牌方在品牌经营平台关联 | **必须先关联，否则创券报 `INVALID_REQUEST: AppID非法`** |
| 回调通知地址 | 调用「设置商品券事件通知地址」API 配置 | HTTPS + 公网可访问，服务商维度配置，所有品牌共用 |

### 核心实体关系

```
品牌（brand_id）← 服务商代操作
└── 商品券（product_coupon_id）
    ├── scope / type / usage_mode
    ├── 【单券 usage_mode=SINGLE】
    │   └── 批次（stock_id）← 同一商品券下最多200个生效中的批次
    │       └── 用户券（coupon_code）
    └── 【多次优惠 usage_mode=PROGRESSIVE_BUNDLE】
        └── 批次组（stock_bundle）
            ├── count: 3-15（可用次数）
            ├── coupon_rule_list: 递进式优惠规则
            └── 用户券（coupon_code）
```

### 发券流程

1. 调用「预发放商品券」API → 获取 token（有效期 5 分钟）
2. 前端使用 token 调起领券组件 → 用户领券
3. 接收回调通知（PRODUCT_COUPON.SEND）
4. 验签（微信支付公钥）+ 解密回调内容
5. 调用「确认发放」API → 券状态变为 EFFECTIVE

### 投放计划（摇一摇场景）

`POST /v3/marketing/partner/delivery-plan/delivery-plans`

- `reuse_coupon_config=true` 可复用商品券和批次信息
- `total_count` 需 ≥ 10000，投放持续时间 ≥ 5 天
- 配置到投放计划的批次，用户通过摇一摇领券由微信侧发放，无需调发券API，但**仍需调确认发放API**

### 用户券状态流转

```
发券 → CONFIRMING（待确认）
        ├── 调用确认API
        │   ├── 未到可用时间 → PENDING → 到达可用时间 → EFFECTIVE
        │   └── 已在可用时间 → EFFECTIVE
        ├── EFFECTIVE → 核销 → USED
        ├── EFFECTIVE → 过期 → EXPIRED
        ├── EFFECTIVE → 用户删除 → DELETED
        └── EFFECTIVE → 品牌失效 → DEACTIVATED
```

## 二、高频踩坑点

### 创建商品券踩坑

`POST /v3/marketing/partner/product-coupon/product-coupons`

- **券选型易混淆**：全场券（scope=ALL）适合拉新推广，单品券（scope=SINGLE）适合爆品精准营销；不确定时推荐从全场满减券开始
- **优惠字段易搞反**：全场券优惠在 `single_usage_info`，单品券在 `stock.single_usage_rule`，搞反会报 INVALID_REQUEST
- 单券用 `stock` 字段，多次优惠用 `stock_bundle` 字段
- 服务商模式**必须在请求体传 `brand_id`**
- 批次的开始时间与结束时间间隔必须 > 5天，否则无法创建投放计划；同一商品券最多 200 个生效中的批次
- 商品券图片（`image_url`）**仅支持通过图片上传 API 获取的 URL**，不可用外部链接
- 商品券详情图：JPG/PNG ≤ 2M，宽高比 65:77（推荐 1170×1326）；图标：透明 PNG ≤ 2M，1:1（建议 1080×1080，留 120px 透明边距）
- 商品券名称等文本字段支持 emoji，按 1 个字符计入长度限制
- `coupon_code_mode` 为 UPLOAD 时需先调预上传券 Code 接口；无自定义券码需求用默认 WECHATPAY 模式即可

### 发券踩坑

- **无论哪个渠道发券（API / 摇一摇 / 领券组件），都必须调用确认发放API**
- 未确认发放前**不要向用户展示"券已到账"**

### 核销踩坑

`POST /v3/marketing/partner/product-coupon/users/{openid}/coupons/{coupon_code}/use`

- **必须关联真实订单**：`associated_order_info`（微信支付订单）或 `associated_pay_score_order_info`（支付分订单），二选一必传
- 核销前券必须为 `EFFECTIVE` 状态（未确认发放的券无法核销）
- `use_time` 格式 RFC3339，`out_request_no` 6-40字符（幂等标识，必须做幂等处理）
- **必须传 `brand_id`**
