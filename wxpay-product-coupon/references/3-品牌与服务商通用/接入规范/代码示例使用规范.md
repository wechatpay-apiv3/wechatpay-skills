# 代码示例使用规范

> 本规范适用于使用本 skill 提供代码示例时的所有场景。

## 核心原则：只检索、不生成

所有代码必须从本 skill 的代码示例文件中检索获取，严禁从零编写任何代码。

**示例代码的定位**：代码示例是伪代码性质，用于讲解业务结构和 API 调用流程，不能直接复制运行。向用户提供代码时，需明确提醒：示例仅展示接口调用结构和签名流程，用户需结合自身项目框架、依赖管理和配置方式进行适配后才能运行。

## 允许的操作

- 从代码示例文件中**读取完整代码**并原样提供给用户
- **替换初始化参数**：`brand_id`、`mchid`、证书序列号、私钥路径、微信支付公钥ID、微信支付公钥路径、回调地址等配置值
- 根据用户实际场景**调整业务参数**：商品券名称、面额、有效期、商品信息等创建券时的业务字段
- 对代码中已有的逻辑添加**注释说明**

## 输出代码时的必要提醒

每次向用户提供代码示例时，必须在代码前或后附带以下提醒：

> ⚠️ 以下代码为结构示例，用于展示接口调用流程和签名逻辑，不能直接复制运行。请结合您的项目框架、依赖管理和实际配置进行适配。

## 禁止的操作

> ⚠️ 违反以下任何一条都可能导致用户接入失败，因为自行编写的代码在参数结构、字段名称、API路径、签名算法等细节上极易出错。

- ❌ **禁止**从零编写 API 调用代码、构造请求体、拼接签名逻辑
- ❌ **禁止**凭记忆或推理编写业务代码，即使用户明确要求"帮我写一个xx"
- ❌ **禁止**拼接或修改示例中的 API 路径、请求方法、Header 结构、签名算法
- ❌ **禁止**将多个示例文件的代码片段拼接组合成新代码
- ❌ **禁止**为示例中未覆盖的语言（如 Python、Node.js）编写等价实现

## 无匹配示例时的处理

当用户需求无法匹配到现有代码示例时（如使用了未覆盖的语言、或需要示例中不存在的 API 操作），**不要尝试自行编写**，而是：

1. 明确告知用户"当前代码示例仅覆盖 Java 和 Go，暂无该语言的示例"
2. 推荐用户参考现有的 Java/Go 示例理解调用逻辑后自行实现
3. 提供对应的 API 接口文档链接供用户参考：
   - 品牌 → [📄 品牌/接入指南/API接口清单.md](../../1-品牌/接入指南/API接口清单.md)
   - 服务商 → [📄 服务商/接入指南/API接口清单.md](../../2-服务商/接入指南/API接口清单.md)

## 交互方式约束

> 向用户确认信息时，须遵守 SKILL.md「全局交互规范」中的两条规则：禁止使用按钮选择组件，所有问题必须得到用户明确回答后才能继续。

## 定位代码示例的步骤

> ‼️ **严格按步骤顺序执行，禁止跨步骤提问。** 每个步骤只确认该步骤要求的信息，禁止在当前步骤中提前询问后续步骤的问题。例如：步骤1只问接入模式和语言，不能同时问接口名称或券类型。
>
> ‼️ **每次用户提出新的接口需求时，必须重新从当前用户消息中提取目标接口，独立执行定位流程。** 严禁将上一轮对话中已定位的接口名称沿用到新请求中。例如：上一轮定位到"查询用户商品券详情"，用户新消息说"我要发券代码"，必须重新识别为「商品券发放」模块，而不是继续使用"查询用户商品券详情"。

### 步骤1：确认接入模式和开发语言

向用户确认两个信息，选择对应的代码目录：
- **接入模式**：品牌直连 / 服务商
- **开发语言**：Java / Go

> ⚠️ 开发语言**必须向用户确认**，禁止根据项目文件或上下文自行推断。
>
> ⚠️ 本步骤**只确认以上两个信息**。禁止在此步骤中同时询问具体接口、券类型（适用范围/优惠类型/使用模式）等后续步骤的内容。
>
> ‼️ **券类型三要素（适用范围、优惠类型、使用模式）只有「创建商品券」接口才需要确认。** 用户请求查询、核销、发券、退券、失效、批次管理等其他任何接口时，**绝对不要在本步骤或任何步骤中询问券类型**。

### 步骤2：定位模块和接口

读取对应的接口索引.md，按「**先定位模块，再定位接口**」的原则匹配。每次只输出一个接口的代码示例。

根据用户输入的明确程度，按下表处理：

| 用户输入类型 | 示例 | 处理方式 |
|---|---|---|
| **指定了具体接口名称** | "核销接口代码"、"查询用户商品券详情" | 直接定位到对应模块和文件，**跳过步骤3，直接进入步骤4输出代码** |
| **关键词可唯一映射到接口** | "创批次"、"查批次"、"上传券码"、"退券" | 确认模块归属后直接定位，无需二次选择，**跳过步骤3，直接进入步骤4输出代码** |
| **提到模块名称或功能类别** | "批次管理"、"券管理"、"发券" | 匹配到模块后，**列出该模块下所有接口让用户选择** |
| **模糊或宽泛** | "给我代码"、"接入代码" | **列出所有可用模块和接口，让用户指定** |

**唯一映射的判断标准**：关键词包含明确的「操作动词 + 操作对象」，且在该模块下只有一个接口与之匹配。

> ‼️ **列出接口时，必须使用接口索引.md 中的原始接口说明文字，禁止自行编造、改写或添加额外的补充描述。** 错误的补充描述可能误导用户对接口用途的理解。例如：接口索引中写的是"确认发放用户商品券"，就只展示"确认发放用户商品券"，不要自行添加"配合预发放使用"等不准确的说明。

> ⚠️ **常见易错场景**：
> - "发券"→ 属于「商品券发放」模块，但该模块下有 3 个接口（发放 / 预发放 / 确认发放），**不能唯一映射**，必须列出让用户选择
> - "创批次"→ 属于「批次管理」模块下的"创建商品券批次"，不要误判为「商品券管理」模块下的"创建商品券"
>
> ‼️ **关键判断：是否需要进入步骤3？**
> - 定位到的接口是「商品券管理」模块中的「创建商品券」→ **需要**进入步骤3确认券类型三要素
> - 定位到的接口是**其他任何接口**（查询、核销、发券、退券、失效、批次管理、回调配置、图片上传等）→ **直接跳到步骤4输出代码，不要询问券类型**

### 步骤3：创建商品券的额外确认（⚠️ 仅限「创建商品券」这一个接口）

> ‼️ **只有「商品券管理」模块中的「创建商品券」这一个接口需要执行本步骤。**
>
> ‼️ **以下接口一律跳过本步骤，直接进入步骤4：** 修改商品券、查询商品券详情、失效商品券、创建批次、查询批次、修改批次、失效批次、关联门店、上传券码、发放商品券、预发放商品券、确认发放、核销、查询用户商品券详情、查询用户商品券列表、失效用户商品券、退券、设置通知地址、获取通知地址、上传图片等所有非「创建商品券」的接口。
>
> ‼️ **自检规则**：在执行步骤3前，必须确认当前接口名称是否**精确等于**「创建商品券」。如果不是，立即跳到步骤4。

**仅当**定位到「商品券管理」模块中的「创建商品券」接口时，需额外确认券类型三要素：

| 要素 | 选项 |
|---|---|
| 适用范围 | 全场（All）/ 单品（Single） |
| 优惠类型 | 满减（Normal）/ 折扣（Discount）/ 兑换（Exchange） |
| 使用模式 | 单券 / 多次优惠（Sequential/Many） |

> ⚠️ 三个要素都**必须向用户确认**，不能从业务描述中自行推断。

> ‼️ **品牌直连与服务商的创建商品券文件命名规则不同，定位文件时必须按对应模式的命名规则查找：**
>
> | 接入模式 | 命名规则 | 单券示例 | 多次优惠示例 |
> |---------|---------|---------|------------|
> | **品牌直连** | `{范围}{类型}[Sequential]{语言}` | `AllNormalJava.java` | `AllNormalSequentialJava.java` |
> | **服务商** | `{范围}{类型}{Single/Many}{语言}` | `AllNormalSingleJava.java` | `AllNormalManyJava.java` |
>
> - 品牌直连：**无后缀** = 单券，带 `Sequential` = 多次优惠
> - 服务商：`Single` 后缀 = 单券，`Many` 后缀 = 多次优惠
> - 详细文件列表见各接口索引.md

### 步骤4：输出代码

读取目标文件的完整代码，将初始化参数替换为用户的实际值。

## 公共库输出策略

业务代码依赖两个公共库文件（SDK工具类 + HTTP客户端）。输出代码时，**不要一次性把公共库代码全部展示给用户**，而是采用分步交互方式：

1. **首次输出**：只展示业务代码（如 `SingleDiscountJava.java`），同时告知用户该代码依赖以下两个公共库文件，需要集成到项目中才能运行：
   - SDK工具类（签名、验签、回调解密）
   - HTTP客户端（请求/响应模型、枚举、HTTP调用）
2. **用户主动索取时**：当用户明确要求查看公共库代码时，再从下方索引中读取并提供对应的完整代码

这样做的目的是避免一次性输出过多代码，让用户先聚焦业务逻辑，再按需获取公共库。
