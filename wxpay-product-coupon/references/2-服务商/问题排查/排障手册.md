# 服务商排障手册

> **错误码提取**：Request-ID 的格式为 `{请求标识}-{错误码}`，**截取最后一个 `-` 后面的数字即为错误码**。例如 `08E68E97CC0610DF0618AC86BCA80120082880CF05-268578704` → 错误码 `268578704`。
>
> **使用方式**：提取到错误码后，先在下方速查表中匹配错误码，命中后查找对应的详细排查章节。未命中则按「常见错误场景」兜底处理。
>
> ‼️ **示例代码推荐输出规则**：各错误码的「示例代码推荐」列出了 Java 和 Go 两种语言的示例，向用户输出时需先确认开发语言，**只输出匹配的示例**。**不要在排障流程一开始就问开发语言**，仅在推荐示例代码时确认即可。
>
> ‼️ **语气要求**：排障回复应亲和、专业，像一个有经验的技术支持在帮忙。避免冷冰冰地罗列信息或堆砌文档目录。用自然的对话语气解释问题原因和修复方案。

## 常见错误场景（兜底）

> 当错误码不在下方速查表中时，按以下场景分类排查。

**参数格式问题**：时间格式不是 RFC3339（正确：`2025-08-01T00:00:00+08:00`）、金额单位错误（应为分）、`out_request_no` 长度不在 6-40 字符、必填字段缺失、`brand_id` 未传 → 对照接口文档逐一校验。

**业务配置问题**："AppID非法" → 未通过交易连接名片关联品牌；"OpenID不合法" → OpenID 与 AppID 不匹配；"全场券/单品券配置错误" → 全场券优惠在 `single_usage_info`，单品券在 `stock.single_usage_rule`。

**签名验签问题**：服务商用 `WECHATPAY2-SHA256-RSA2048` + `mchid="xxx"`。详细自查清单见 [📄 签名验签规范与排查.md](../../3-品牌与服务商通用/接入规范/签名验签规范与排查.md)。

**权限问题**：检查品牌方是否已授予服务商「商品券」产品功能权限；确认 brand_id 与当前身份匹配。

**频率限制**：降低请求频率，使用指数退避策略重试（1s → 2s → 4s）。服务商代理多品牌时各品牌共用频率配额，需做全局限流。

**服务端异常**：使用相同的 `out_request_no` / `send_request_no` 重试（幂等），指数退避，最多重试 3 次。

## 一、参数校验类

### 268435461 — 参数非法

**场景**：请求参数值格式或内容不合法。

**排查步骤**：
1. 检查时间格式是否为 RFC3339（正确示例：`2025-08-01T00:00:00+08:00`）
2. 检查金额字段单位是否为**分**（非元）
3. 检查枚举值是否拼写正确（如 `ALL`/`SINGLE`、`NORMAL`/`DISCOUNT`/`EXCHANGE`）
4. 检查 URL 格式字段是否以 `https://` 开头
5. 检查是否有不可见字符（如全角空格、BOM 头等）
6. 检查 `brand_id` 是否已填写，服务商模式下 `brand_id` 为必填参数

**修复建议**：对照接口文档逐一校验参数格式，特别注意时间格式和金额单位。

---

### 268578526 — 解析 OpenID 错误

**场景**：请求中的 `openid` 参数格式不合法或与 `appid` 不匹配。

> ⚠️ 注意：服务商的错误码是 268578**526**，与品牌直连的 268578**532** 不同。

**排查步骤**：
1. 确认 `openid` 格式是否正确（以 `o` 开头的 28 位字符串）
2. 确认 `openid` 是否对应请求中的 `appid`（OpenID 与 AppID 是一一对应的）
3. 确认 `appid` 是否已通过服务商授权关联品牌
4. 检查是否混用了不同 AppID 下的 OpenID
5. 确认是否传了品牌方的 AppID 而非服务商自身的 AppID

**修复建议**：确保 `openid` 和 `appid` 来自同一个微信应用，且服务商已获得品牌方授权。

---

### 268578101 — 入参不合法

**场景**：请求参数结构不合法，通常是必填字段缺失或请求体格式错误。

**排查步骤**：
1. 检查请求体是否为合法的 JSON 格式
2. 检查所有必填字段是否已填写（服务商模式下 `brand_id` 为必填）
3. 检查字段名是否拼写正确（如 `product_coupon_id` 不要写成 `coupon_id`）
4. 检查嵌套对象结构是否正确（如 `stock.stock_send_rule` 不要写成扁平结构）

**修复建议**：对照接口文档检查请求体结构，确保必填字段完整、嵌套层级正确。

---

### 268435464 — 参数超出取值范围

**场景**：请求参数值超出了允许的数值范围。

**排查步骤**：
1. 检查金额类字段是否在合理范围内（如 `discount_amount` 不能大于 `threshold`）
2. 检查数量类字段（如 `max_count`、`max_count_per_user`）是否超出上限
3. 检查折扣类字段（如 `percent_off`）取值范围（通常 1-99）
4. 检查多次优惠的 `count` 是否在 3-15 范围内

**修复建议**：调整参数值至允许的范围内，对照接口文档确认各字段的取值约束。

---

### 268578224 — 参数错误，请根据接口文档检查字段要求

**场景**：通用的参数校验失败，通常是字段值不满足业务规则约束。

**排查步骤**：
1. 检查请求返回的详细错误信息（`detail` 字段），通常会指明具体哪个字段有问题
2. 常见情况：
   - 批次的开始时间与结束时间间隔必须大于 5 天
   - 全场券优惠配置在 `single_usage_info`，单品券配置在 `stock.single_usage_rule`，不要搞混
   - 多次优惠的 `count` 必须在 3-15 之间
3. 对照接口文档逐一校验字段约束

**修复建议**：根据详细报错信息定位具体字段，对照接口文档修正。

## 二、券状态与生命周期类

### 268578776 — 用户券已过期

**场景**：对已过期的券执行核销、退券等操作时触发。

**常见真实原因**：

1. **有效期起算时间不一致**：微信侧以**回调通知时间**作为券的生效起算时间，但商家侧/服务商侧以发券接口返回成功的时间作为生效时间。两边起算时间有差异时，商家侧认为券还在有效期内，但微信侧已判定过期。
   - 正确做法：**记录收到回调通知的时间**，确认发放时以回调时间为准计算有效期。
2. **确认发放延迟导致有效期缩短**：用户在微信侧摇到券后，商家因业务流程（如要求用户先注册会员）延迟调用确认发放接口。延迟期间有效期已在消耗。
   - 典型场景：用户摇到券 → 商家要求注册会员 → 用户犹豫未注册 → 几天后注册并确认发放 → 券有效期已过大半或已过期。
3. **确认发放后商家侧/服务商侧单方面修改有效期**：确认发放时已将有效期同步给微信（如 3 天），之后在自己系统里改为 5 天，但微信侧仍按原值计算。
   - 关键点：**确认发放后券的有效期无法修改**，微信侧没有修改券有效期的接口。

**排查步骤**：

> 📋 **排查前需要你提供以下信息**：
>
> **身份配置参数：**
> - **服务商商户号**（mchid）
> - **API 证书序列号**（serial_no）
> - **微信支付公钥 ID**（wechat_pay_public_key_id）
> - **签名值（Base64）**+**时间戳**+**随机串**
>
> **业务参数：**
> - **品牌 ID**（brand_id）
> - **券码**（coupon_code）—— 出问题的那张券
> - **用户 OpenID**（openid）
> - **商品券 ID**（product_coupon_id）
> - **批次 ID**（stock_id）
> - **AppID**（appid）
>
> ‼️ **所有参数均为必填，必须收齐后才能执行脚本，严禁对缺失参数使用默认值或自行推断。** 身份配置参数在同会话中首次收集后可复用。
>
> ⚠️ **不要在第一步就列出参数清单**，避免一上来就堆信息，影响用户体验。

1. 收齐上述参数后，执行排障脚本 [`查询用户商品券详情_服务商.py`](../../scripts/查询用户商品券详情_服务商.py)（脚本会自动构造请求、发送并结构化输出关键字段和诊断提示）。具体参数收集和脚本调用流程见 [SKILL.md](../../SKILL.md)「5.1 排障脚本」。
2. 根据脚本输出的「券关键信息」和「自动诊断」确认过期原因：重点关注 `券状态`、`有效期结束` 时间、`领取时间` 与 `确认发放时间` 的间隔
3. 对比输出的时间信息与商家侧/服务商侧记录的有效期是否一致
4. 检查商家侧是否在确认发放后单方面修改了有效期配置

> 💡 **示例代码推荐**：排查此问题需要查询券状态，**必须主动向用户推荐以下示例代码**：
> - 查询用户商品券详情：
>   - Java 示例：`2-服务商/代码示例/Java/5-商品券查询/GetUserProductCoupon.java`
>   - Go 示例：`2-服务商/代码示例/Go/5-商品券查询/GetUserProductCoupon.go`

**修复建议**：

1. **当前这张过期券**：已是终态，无法再执行核销、退券等任何操作。业务侧应将该券状态同步为「已过期」，并向用户说明券已失效。
2. **预防后续券再出现意外过期**：
   - **有效期起算时间必须以回调通知时间为准**，在收到回调时记录时间戳，不要以发券接口返回时间作为起算点
   - **收到发券回调后尽快调用确认发放接口**，减少延迟导致的有效期损耗
   - **确认发放后不要在商家侧/服务商侧单方面修改有效期**，微信侧没有修改券有效期的接口，如需调整须创建新批次

### 268578367 — 用户券状态非法

**场景**：对券执行的操作与当前券状态不匹配。

**常见真实原因**：

1. **没调确认发放就直接核销**：收到发券回调后跳过了确认发放，直接调核销接口。此时券状态还是 `CONFIRMING`，核销要求 `EFFECTIVE`，状态不匹配报错。
2. **券已失效后还来调核销**：券已被失效（`DEACTIVATED`），商家侧/服务商侧没同步状态，还对该券发起核销。

**排查步骤**：
1. 调用「**查询用户商品券详情**」接口查看当前券状态。向用户收集排查所需的参数，然后执行排障脚本 [`查询用户商品券详情_服务商.py`](../../scripts/查询用户商品券详情_服务商.py)（具体流程见 [SKILL.md](../../SKILL.md)「5.1 排障脚本」）。
2. 根据脚本输出的 `券状态` 判断：如果是 `CONFIRMING`，说明缺少确认发放步骤
3. 如果是 `DEACTIVATED`，说明券已失效，需同步状态

> 💡 **示例代码推荐**：根据排查结果，用户可能需要接入以下接口，**必须主动向用户推荐以下示例代码**：
> - 查询用户商品券详情：`2-服务商/代码示例/Java/5-商品券查询/GetUserProductCoupon.java`
> - 确认发放用户商品券：`2-服务商/代码示例/Java/3-商品券发放/ConfirmSendUserProductCoupon.java`
> - 核销用户商品券：`2-服务商/代码示例/Java/4-商品券核销/UseUserProductCoupon.java`

**修复建议**：
- 必须按 发券回调 → **确认发放** → 核销 的流程走，不能跳步
- 对已失效的券不再发起操作

### 272756769 — 券已被删除

**场景**：操作的券已被删除。

**常见真实原因**：

1. **用户在微信卡包中手动删除了券**：用户可以在微信卡包中主动删除券，目前**仅卡包路径可以触发删除**，且**没有回调通知机制**——微信不会向商户/服务商推送"券被用户删除"的通知。商户/服务商侧完全不知道券已被删除，再对该券发起操作时就会报错。

**排查步骤**：
1. 调用「**查询用户商品券详情**」接口确认券状态。向用户收集排查所需的参数，然后执行排障脚本 [`查询用户商品券详情_服务商.py`](../../scripts/查询用户商品券详情_服务商.py)。
2. 确认脚本输出的券状态为 `DELETED`
3. 确认商户/服务商侧没有主动操作过该券，则为用户卡包删除

> 💡 **示例代码推荐**：建议在核销等关键操作前先查询券状态，**必须主动向用户推荐以下示例代码**：
> - 查询用户商品券详情：
>   - Java 示例：`2-服务商/代码示例/Java/5-商品券查询/GetUserProductCoupon.java`
>   - Go 示例：`2-服务商/代码示例/Go/5-商品券查询/GetUserProductCoupon.go`

**修复建议**：
- 已删除的券无法恢复，业务侧需更新本地券状态，不再对该券发起操作
- **在核销等关键操作前，先查询券详情确认券状态**，防止因用户卡包删券而操作失败
- 注意：用户卡包删券目前无回调通知，商户/服务商侧无法实时感知，只能在操作前主动查询

### 268577984 — 商品券批次状态不是发放中

**场景**：尝试对一个非「发放中」状态的批次执行发券操作。

**排查步骤**：
1. 调用「查询商品券批次详情」接口，确认批次状态（`state`）
2. 批次状态对照：
   - `AUDITING` — 审核中，需等待审核通过
   - `SENDING` — 发放中（仅此状态可发券）
   - `PAUSED` — 已暂停，需恢复后才能发券
   - `STOPPED` — 已停止
   - `DEACTIVATED` — 已失效
3. 检查批次的有效期是否已过（过期的批次会自动停止）

## 三、发券与核销限制类

### 272756740 — 用户领取券张数已达上限

**场景**：用户领取同一批次的券数量已达到 `max_count_per_user` 的限制。最常见于私域塞券场景，创券限领 N 张但发券侧没做计数校验导致超发。

**排查步骤**：
1. 查批次配置的 `max_count_per_user`（必填，最多 100 次）
2. 查该用户在此批次下已有多少张券（含未确认发放的也计入名额）

> 💡 **示例代码推荐**：排查此问题需要查询用户已领券数量和批次配置，**必须主动向用户推荐以下示例代码**：

**修复建议**：发券前先查用户已领张数，达上限不再发券。需要再发的话增大 `max_count_per_user` 或创新批次。

### 272755200 — 发券请求重入但单号不一致

**场景**：对同一用户、同一批次发起了重复的发券请求，但两次请求使用了不同的 `out_request_no`。

**排查步骤**：

### 268578927 — 使用了相同单号，但与上次请求的数据不一致

**排查步骤**：
1. 查找首次请求的完整请求体（通过日志或 Request-Id）
2. 比对两次请求的所有字段，找出不一致的参数

**修复建议**：幂等重试时，`out_request_no` 和请求体必须完全一致。如果需要修改参数，必须使用新的 `out_request_no`。

## 四、业务配置类

### 268578704 — 传入的 AppID 非法

**场景**：请求中的 `appid` 未与品牌建立关联，或服务商未获得该品牌的授权。

**排查步骤**：
1. 确认 `appid` 是否正确（注意大小写）
2. 确认该 AppID 是否已关联品牌
3. 确认服务商是否已获得该品牌的「商品券」产品功能授权

**修复建议**：确认品牌授权和 AppID 关联状态。参考 [📄 MAB关系说明文档.md](../../3-品牌与服务商通用/接入规范/MAB关系说明文档.md)。

### 268578608 — 当前访问对象不存在，请确认 ID 是否合法、属于当前品牌且由当前服务商创建

**场景**：请求中的资源 ID 不存在、不属于当前品牌、或不是由当前服务商创建。

**排查步骤**：
1. 确认 `product_coupon_id` / `stock_id` 是否拼写正确
2. 确认该资源是否属于请求中的 `brand_id`
3. 确认该资源是否由**当前服务商**创建（服务商 A 创建的资源，服务商 B 无法操作）
4. 确认该资源是否已被失效或删除

**修复建议**：核实资源 ID 和归属关系，服务商只能操作自己创建的资源。

### 268579803 — 服务商没有此接口权限

**场景**：服务商调用了未获授权的接口，或品牌方未给该服务商开通「商品券」产品功能权限。

**排查步骤**：
1. 确认品牌方是否已在品牌经营平台授予服务商「商品券」产品功能权限
2. 确认服务商的 `mchid` 是否正确
4. 确认是否使用了正确的 API 路径前缀（服务商应为 `/v3/marketing/partner/`）

**修复建议**：联系品牌方在品牌经营平台完成对服务商的授权。

## 五、频率与系统类

### 270924289 — 频率过快

**场景**：接口调用频率超过限制。

**排查步骤**：
1. 检查短时间内是否有大量并发请求
2. 确认是否有重试逻辑导致请求放大
3. 如果服务商代理多个品牌，各品牌的请求会共用服务商的频率配额

**修复建议**：
- 降低请求频率，使用**指数退避策略**重试（第1次等1秒、第2次等2秒、第3次等4秒）
- 对批量操作做限流控制
- 服务商代理多品牌时需做全局限流
